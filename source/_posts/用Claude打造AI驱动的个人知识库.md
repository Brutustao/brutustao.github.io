## 一、缘起：知识管理的老问题

在信息过载的时代，我们每天都在消费大量内容——技术文章、教程视频、行业报告、播客访谈。但多数人面临同样的问题：**看过就忘，想找找不到，知识散落各地**。

传统的文件管理方式（按文件夹分类）在面对几百上千条笔记时迅速失效。标签体系看似灵活，久了也会变得混乱。我们需要一种既能**沉淀知识**，又能**让 AI 参与管理**的知识库结构。

## 二、底座：PARA 方法 + Obsidian

一切从框架开始。我选择 [PARA 方法](https://fortelabs.com/blog/para/)（Projects, Areas, Resources, Archives）作为顶层分类逻辑：

```
知识库/
├── 1-项目/     # 有明确截止日期的任务
├── 2-领域/     # 持续关注的领域（网站建设、编程等）
├── 3-资源/     # 感兴趣的主题资料
├── 4-归档/     # 已完成的项目
├── 5-日记/     # 每日记录
├── 6-模板/     # 笔记模板
└── 附件/       # 图片等资源
```

PARA 的核心智慧在于区分"项目"和"领域"：项目有终点，领域没有终点。学一门课程是项目（会结束），但关注"AI 工程化"是领域（持续积累）。这种区分让知识库既有行动力，又有沉淀感。

工具层面选择 **Obsidian**，因为它基于纯文本 Markdown、支持双向链接，且生态丰富。更重要的是——AI 工具能直接读写 Markdown 文件，这是知识库"可编程"的前提。

## 三、进化：从文件夹到 Wiki 模式

纯 PARA 结构有一个问题：**知识散落在各自文件夹里，缺少横向串联**。比如一篇关于 RLHF 的笔记放在"对齐与微调"下，一篇关于 GPT 的笔记放在"语言模型"下，它们之间的关联只能靠人脑记忆。

解决方式是引入 **Wiki 模式**，核心三件套：

### 1. 知识卡片替代零散笔记

每条知识不再是一个无结构的自由笔记，而是使用**统一模板**的"知识卡片"：

```yaml
---
date: 2026-05-12
type: 知识卡片
tags: [知识卡片, 领域标签]
aliases: [相关关键词]
status: 待完善
---

# 标题

## 一句话总结

## 核心内容

## 与其它笔记的联系
- [[相关笔记]] → 关联说明
```

模板的关键在于：**强制结构化**。一句话总结迫使你提炼核心，关联关系要求你主动连接已有知识。

### 2. MOC 索引代替文件夹导航

MOC（Map of Content）是一篇**索引笔记**，汇总某一领域下的所有知识卡片。它不是文件夹那种"物理隔离"，而是可以跨目录聚合的逻辑视图：

```
📚 资源 MOC
├── 经验总结
│   └── 用 Claude 搭建 Obsidian 知识库
├── AI 工具与方法论
│   ├── Claude Code Skill 系统
│   └── Agent Skill 规范与设计模式
└── 外部关联
    └── MOC-领域 → 关注领域索引
```

有了 MOC，"浏览知识库"变成"浏览 MOC"，比翻文件夹高效得多。

### 3. 双向链接织网

每条知识卡片末尾的"与其它笔记的联系"不是装饰，而是知识网的节点。例如 LLM 智能体笔记链接到 GPT 系列、Transformer 架构、RLHF 原理，反过来这些笔记也回链到智能体。**信息不再是孤岛，而是网络。**

## 四、点睛：Claude 驱动的工作流

结构搭好后，最大的痛点是维护成本。每篇笔记都需要：确定分类、填写模板、搜索关联笔记、更新 MOC 索引、检查断链——手工做非常繁琐。

解决方案是用 **Claude Code** 驱动三个工作流：

### /ingest — 知识摄取

当我说"摄取 LLM 智能体"，Claude 自动完成：
1. 搜索互联网获取核心信息
2. 用知识卡片模板创建笔记
3. 放入正确的目录
4. 查找已有笔记并建立双向链接
5. 更新对应 MOC 索引
6. 标记为"待完善"

整个过程从"自己动手 15 分钟"变成"一句话的事"。

### /query — 查询学习

遇到问题时不翻阅文件，直接问。Claude 先搜索本地笔记，若内容不足再从互联网补充，返回答案并注明引用来源。

### /lint — 巡检维护

知识库需要定期维护，但人工检查几十篇笔记的断链和未注册条目很痛苦。/lint 自动检查：所有链接是否指向存在的文件、MOC 中注册的笔记是否存在、是否有笔记未注册到 MOC。

## 五、扩展：从专区到全库

最初这套模式只用在"卡帕西 LLM 知识专区"（一个专门的知识子库）。但效果太好了，于是推广到整个知识库：

- **通用知识卡片模板** — 不限于 LLM，适用于任何主题
- **各区域 MOC** — 领域有 MOC-领域，资源有 MOC-资源
- **统一入口** — 欢迎页汇总所有 MOC，一站式导航

现在无论摄取 AdSense 机制、Skill 系统，还是任何新主题，流程都一样：确定所属区域 → 创建知识卡片 → 注册到 MOC → 建立关联。

## 六、经验与教训

**什么做对了：**

1. **结构先行** — 先搭好 PARA 框架和模板再填内容，避免后期重构
2. **CLAUDE.md 是关键** — 把所有工作流的指令写进 CLAUDE.md，每次对话 Claude 自动加载，保持行为一致
3. **双向链接不留死角** — 每篇笔记必关联至少 2 篇已有笔记，长期积累下来的网络效应远超单篇笔记的价值
4. **模板驱动** — 用模板约束格式，确保 AI 能稳定解析和生成

**可以改进的：**

1. **教程笔记需人工填充** — AI 可以抓取要点摘要，但"我的理解"部分需要亲自观看视频/阅读后补充
2. **知识卡片深度有限** — 当前卡片偏向概览，真正深入需要持续迭代
3. **MOC 需要维护** — 随着笔记增多，MOC 可能变得臃肿，需要二级分类或子 MOC

## 七、给你的建议

如果你想在自己的知识库中复现这套模式，建议的分步路径：

1. 第一天：搭建 PARA 文件夹结构 + Obsidian 基础配置
2. 第二天：创建 3-5 篇核心知识卡片，熟悉模板
3. 第三天：建立第一个 MOC，把卡片串联起来
4. 第四天：配置 CLAUDE.md，打通 /ingest /query /lint
5. 持续：每天用 /ingest 摄取新知识，每周用 /lint 巡检一次

最关键的不是工具，而是**把知识管理当成系统来设计**——有结构、有流程、有自动化。当 AI 承担了分类、链接、检查的重复劳动，你就能专注于真正有价值的事情：思考和理解。
